# Todo

## Plan
Build the async web framework top-down, starting with the core Sanic application class and routing (the most user-facing feature), then layering on request/response handling, middleware, blueprints, configuration, class-based views, cookies, exception handling, signals, static file serving, and WebSocket support. Each task delivers a complete, testable feature with its required infrastructure.

## Tasks
- [x] Task 1: Implement the core web application with route registration and basic request/response handling. Users create an app instance, register routes using decorators (@app.get, @app.post, etc.), and handlers receive a request object and return a response. Support path parameters with type conversion (string, int, float), query string parsing, and response helpers (json, text, html, raw, redirect, empty). Include a test client for making requests against the app without starting a real server.
- [x] Task 2: Implement middleware support so users can register functions that run before every request or after every response. Middleware can modify the request, short-circuit the response, or modify the response. Support priority-based ordering so users control execution order. Provide @app.on_request and @app.on_response decorator shortcuts.
- [x] Task 3: Implement blueprints for modular route organization. Users create Blueprint objects with their own routes, middleware, and exception handlers, then register them on the app. Blueprints support URL prefixes, grouping multiple blueprints together, and nested blueprint composition. Blueprint middleware only applies to that blueprint's routes.
- [x] Task 4: Implement the configuration system so users can set app settings via a dict-like Config object. Support loading config from environment variables (with a configurable prefix like SANIC_), from Python files, from dicts, and from objects. Provide sensible defaults for common settings like request timeouts, max request size, and keep-alive.
- [x] Task 5: Implement custom exception handling so users can register handlers for specific exception types. When a route raises an exception, the framework finds the most specific registered handler and calls it. Provide built-in HTTP exceptions (NotFound, BadRequest, Unauthorized, Forbidden, MethodNotAllowed, ServerError) with appropriate status codes and a default error handler that returns JSON or text error responses.
- [x] Task 6: Implement class-based views (HTTPMethodView) so users can define request handlers as classes with get(), post(), put(), patch(), delete() methods. Support applying decorators to all methods and attaching views to routes. The framework dispatches to the correct method based on the HTTP method.
- [x] Task 7: Implement cookie handling for both request parsing and response setting. Parse cookies from incoming request headers into a dict-like object. Provide response methods to add, modify, and delete cookies with support for path, domain, secure, httponly, samesite, max_age, and expires attributes. Handle cookie prefixes (__Host- and __Secure-).
- [x] Task 8: Implement a signal/event system so users can register handlers for lifecycle events (before server start, after server start, before server stop, after server stop). Support custom named signals that users can dispatch and listen to. Provide both decorator-based and method-based registration.
- [x] Task 9: Implement static file serving so users can serve files from a directory. Support configuring the URL prefix, directory path, and content type detection. Handle file-not-found errors and directory index files.
- [x] Task 10: Implement WebSocket support so users can define WebSocket handlers that receive and send messages. Provide a WebSocket route decorator and a WebSocket connection object with send/receive methods. Handle connection lifecycle (open, message, close).
- [x] Task 11: Implement named routes and URL building so users can generate URLs for registered routes by name. Support reverse URL lookup with path parameter substitution. Handle blueprint route names with proper namespacing.
- [>] Task 12: Implement route versioning so users can version their API endpoints. Support integer, float, and string versions with a configurable version prefix (default /v). Routes with version=1 are accessible at /v1/path.
